# BodyJ4You Miami Monday Reorder Dashboard - Build Spec

## Overview

A single-page web app that lets a user upload a Miami Monday Reorder xlsx file
and renders it as a styled dashboard. The xlsx is generated by an upstream
process (the `inventory-reorder` skill in our Claude workflow). The web app's
ONLY job is to read the file and visualize it. It MUST NOT recompute, derive,
filter, or transform any business metric. Every visualization value comes
directly from a cell in the xlsx.

## Hard rules (non-negotiable)

1. **The Excel file is the only source of truth.** The web app reads cells
   and renders them. It does NOT calculate anything вҖ” no velocity, no days,
   no totals, no section assignment. All of those are columns in the xlsx.
2. **No backend required.** This is a static client-side app. File upload
   stays in the browser. No server, no database, no auth.
3. **No data is sent over the network** beyond hosting static assets.
4. **No file persistence.** Refresh = re-upload. (Optional: in-browser
   sessionStorage of last-uploaded file is OK but not required.)

## Tech stack

- **Framework**: React + Vite (TypeScript preferred but JS is fine)
- **xlsx parser**: `xlsx` (SheetJS) вҖ” `npm install xlsx`
- **Styling**: plain CSS or Tailwind, your call. Reference styles already
  defined in the prototype HTML (see "Visual reference" below).
- **Hosting**: any static host (Vercel, Netlify, GitHub Pages, S3+CloudFront)
- **No router needed** вҖ” single page

## Input file structure

The user uploads an xlsx file with this structure. The web app MUST read it
exactly as specified. Do not try to "be smart" about variations.

### Sheet 1: "Miami Reorder"

Row 1 = headers. Rows 2+ = data. One row per SKU.

| Col | Header             | Type    | Notes                                                   |
|-----|--------------------|---------|---------------------------------------------------------|
| A   | Section            | string  | "URGENT FBA" / "PLANNED FBA" / "UV" / "STEEL"           |
| B   | SKU                | string  | Display SKU                                             |
| C   | ASIN               | string  | Amazon ASIN                                             |
| D   | Parent ASIN        | string  | Parent SKU name (can be empty)                          |
| E   | Category           | string  | "PA" / "EO" / "PD" / "HB" / "GK" / etc.                 |
| F   | Available          | number  | Units available at FBA                                  |
| G   | Inbound            | number  | Units in transit to FBA                                 |
| H   | Reserved           | number  | Units reserved (in orders)                              |
| I   | Amazon Days        | number  | Raw days-of-supply from Amazon report                   |
| J   | Display Days       | number  | Pre-computed days to display in the bar                 |
| K   | Out Of Stock       | boolean | TRUE if Available = 0                                   |
| L   | Weighted Velocity  | number  | Daily units (decimal, e.g. 13.66)                       |
| M   | Min Level          | number  | FBA minimum inventory level                             |
| N   | Status             | boolean | TRUE/FALSE вҖ” pipeline >= min level                      |
| O   | QTY                | number  | Recommended ship quantity                               |

### Sheet 2: "Summary"

Row 1 = headers. Rows 2+ = data.

| Col | Header      | Type    | Notes                                       |
|-----|-------------|---------|---------------------------------------------|
| A   | Section     | string  | "URGENT FBA" / "PLANNED FBA" / "UV" / "STEEL" / "GRAND TOTAL" |
| B   | SKU Count   | number  | Number of SKUs in this section              |
| C   | Total Units | number  | Sum of QTY for this section                 |

Always has 5 rows of data: URGENT FBA, PLANNED FBA, UV, STEEL, GRAND TOTAL.
A section with 0 SKUs still gets a row with count=0, total=0.

## App flow

1. **Empty state** (no file uploaded): Show a centered drop zone / upload button.
   "Upload your Miami Reorder xlsx file to view the dashboard." Click or drag.
2. **Parsing**: Read file with SheetJS. Parse both sheets. Validate columns.
3. **Error state** (parse failure, wrong sheets, missing columns): Show error
   message naming the issue. Offer "Try a different file" button.
4. **Loaded state**: Render the dashboard.
5. **"Upload a different file" button** persistent in the corner of the
   loaded state.

### Table

5 column blocks with subtle background tints + a 2px left border between blocks:

| Block     | Columns                          | Tint     |
|-----------|----------------------------------|----------|
| Product   | SKU, ASIN, Parent, Cat           | indigo   |
| Inventory | Available, Inbound, Reserved     | green    |
| Demand    | Days, Velocity                   | red      |
| Reference | Min, Status                      | gray     |
| Reorder   | QTY                              | yellow   |

Section rows: dark navy band spanning all 12 columns, showing
`в—Ҹ URGENT FBA   22 SKUs В· 2,720 units`. The count and total come from the
Summary sheet, NOT computed from data rows.

Within each section, render data rows in the order they appear in the xlsx
(do not re-sort). When the Parent ASIN value changes from the previous row,
add a thin gray top-border to subtly group variants.

### Filter pills

Click a pill to filter visible rows. Filtering hides DOM rows. It does NOT
recompute totals вҖ” stat cards stay the same. Pills toggle, last-clicked wins.

### Days column (the visualization)

For each data row:
- If `Out Of Stock = TRUE` вҶ’ render red "OUT OF STOCK" badge, no bar
- Otherwise вҶ’ render a horizontal bar scaled 0-60 days with the number
  on the right:
  - **Bar fill width** = `min(100, (Display Days / 60) * 100)` percent
  - **Color**: `Display Days < 30` red, `< 45` yellow, `>= 45` green
  - **Tick mark** at 50% (the 30-day midpoint)
  - **Big bold number** to the right of the bar: `{Display Days}d`

Note: the web app reads `Display Days` directly. Do NOT compute it from
Available / Velocity. The xlsx already has the correct value.

### QTY column (the action)

Large bold text. Font size ~20px, weight 800, color amber (`#92400e`),
yellow background tint. This is the number warehouse will execute on.

## File reading code (reference implementation)

```ts
import * as XLSX from 'xlsx';

interface DataRow {
  Section: string;
  SKU: string;
  ASIN: string;
  'Parent ASIN': string;
  Category: string;
  Available: number;
  Inbound: number;
  Reserved: number;
  'Amazon Days': number;
  'Display Days': number;
  'Out Of Stock': boolean;
  'Weighted Velocity': number;
  'Min Level': number;
  Status: boolean;
  QTY: number;
}

interface SummaryRow {
  Section: string;
  'SKU Count': number;
  'Total Units': number;
}

async function parseFile(file: File): Promise<{
  data: DataRow[];
  summary: SummaryRow[];
}> {
  const buffer = await file.arrayBuffer();
  const wb = XLSX.read(buffer, { type: 'array' });

  if (!wb.SheetNames.includes('Miami Reorder')) {
    throw new Error('Missing sheet "Miami Reorder"');
  }
  if (!wb.SheetNames.includes('Summary')) {
    throw new Error('Missing sheet "Summary"');
  }

  const data = XLSX.utils.sheet_to_json<DataRow>(wb.Sheets['Miami Reorder']);
  const summary = XLSX.utils.sheet_to_json<SummaryRow>(wb.Sheets['Summary']);

  // Validate columns
  const requiredDataCols = [
    'Section', 'SKU', 'ASIN', 'Parent ASIN', 'Category',
    'Available', 'Inbound', 'Reserved',
    'Amazon Days', 'Display Days', 'Out Of Stock',
    'Weighted Velocity', 'Min Level', 'Status', 'QTY',
  ];
  if (data.length > 0) {
    const cols = Object.keys(data[0]);
    const missing = requiredDataCols.filter(c => !cols.includes(c));
    if (missing.length) {
      throw new Error(`Miami Reorder sheet missing columns: ${missing.join(', ')}`);
    }
  }

  return { data, summary };
}
```

## Filter logic

```ts
type SectionKey = 'all' | 'URGENT' | 'PLANNED' | 'UV' | 'STEEL';

const SECTION_LABEL_TO_KEY: Record<string, SectionKey> = {
  'URGENT FBA': 'URGENT',
  'PLANNED FBA': 'PLANNED',
  'UV': 'UV',
  'STEEL': 'STEEL',
};

function filterRows(rows: DataRow[], filter: SectionKey): DataRow[] {
  if (filter === 'all') return rows;
  return rows.filter(r => SECTION_LABEL_TO_KEY[r.Section] === filter);
}
```

## Days bar rendering

```tsx
function DaysBar({ days, isOos }: { days: number; isOos: boolean }) {
  if (isOos) {
    return <div className="oos-badge">OUT OF STOCK</div>;
  }
  const pct = Math.min(100, (days / 60) * 100);
  const color = days < 30 ? '#dc2626' : days < 45 ? '#eab308' : '#16a34a';
  return (
    <div className="days-cell-content">
      <div className="days-bar-bg">
        <div className="days-tick" />
        <div className="days-bar-fill" style={{ width: `${pct}%`, background: color }} />
      </div>
      <span className="days-big">{Math.round(days)}d</span>
    </div>
  );
}
```

## Acceptance criteria

The build is correct when:

1. Uploading the provided `Reorder_Miami__05_12_26.xlsx` file renders a
   dashboard that visually matches `Miami_Monday_Viewer.html`
2. Stat cards show: Total 4,060 units / 38 SKUs В· 22 URGENT В· 2 PLANNED В·
   6 UV В· 8 STEEL (all values from Summary sheet)
3. Filter pills correctly hide/show sections
4. Days bar colors match: red <30, yellow 30-45, green 45+
5. OUT OF STOCK badge shows for rows where `Out Of Stock = TRUE`
6. QTY column is visually large (20px+) and amber-colored
7. Uploading a malformed file (wrong sheets, missing columns) shows a
   clear error, not a crash
8. "Swap file" button returns to the empty state

## Non-goals (explicit DO NOT do)

- Do NOT compute velocity from t7/t30/t60/t90
- Do NOT compute display days from Available / Velocity
- Do NOT recalculate Out Of Stock from Available
- Do NOT regenerate the Summary sheet from the data sheet
- Do NOT add "edit" functionality (no in-place quantity overrides)
- Do NOT add export functionality (the user already has the file)
- Do NOT add charts or analytics (this is a table viewer, not a BI tool)
- Do NOT add login, auth, user accounts, or any persistence
- Do NOT validate business logic (the xlsx is trusted as correct)

The web app is a renderer. Future versions might add features, but v1 is
strictly a visualization of the file.
