# Miami Monday Reorder Dashboard

A single-page web app that visualizes a Miami Monday Reorder xlsx file as a styled dashboard. The xlsx is generated by an upstream process (the `inventory-reorder` skill); this app is **strictly a renderer** — it does not compute, derive, or transform any business metric.

Built per `web_dashboard_spec.md` against the visual reference in `Miami_Monday_Viewer.html`.

## Stack

- **React 18 + Vite 5** (JavaScript, no TypeScript)
- **SheetJS (`xlsx`)** for client-side xlsx parsing
- Plain CSS lifted from the prototype
- **No backend, no auth, no persistence** — file stays in the browser

## How it works

1. **Empty state**: drop zone / "Choose file" button.
2. **Parsing**: `parseFile.js` calls `XLSX.read(arrayBuffer)`, requires the two sheets `Miami Reorder` and `Summary`, and validates that every required column is present. Any failure raises a typed error message.
3. **Error state**: shows the error and a "Try a different file" button.
4. **Loaded state**: renders the dashboard.
5. **Swap file** (top-right): returns to the empty state per spec acceptance #8.

### Hard rules (enforced by code)

- **Display Days** is read straight from column J — never computed from Available / Velocity.
- **Out Of Stock** is read straight from column K — never derived from `Available === 0`.
- **Section counts and totals** in the header bar, stat cards, and section rows come from the **Summary** sheet only — never from counting data rows.
- **Filter pills** hide DOM rows. They do **not** recompute stats.

### Days bar

For each non-OOS row, the bar fill is `min(100, (Display Days / 60) * 100)%` with color thresholds:

- `< 30` → `#dc2626` (red, critical)
- `< 45` → `#eab308` (yellow, OK)
- `>= 45` → `#16a34a` (green, goal)

A tick mark sits at 50% (the 30-day midpoint) and the rounded day count appears in bold next to the bar.

OOS rows (Out Of Stock = TRUE) render an `OUT OF STOCK` badge with no bar.

## Deploy notes

The app is a pure static SPA. Any of these work without configuration:

- **Vercel**: `vercel deploy` from this folder. Framework auto-detected as Vite.
- **Netlify**: drag `dist/` into the dashboard, or `netlify deploy --dir=dist --prod`.
- **GitHub Pages**: `npm run build` then push `dist/` to a Pages branch. Note `base: './'` in `vite.config.js` is set so relative paths work under a sub-path.
- **S3 + CloudFront**: `aws s3 sync dist/ s3://your-bucket/` and set CloudFront default root to `index.html`.

No environment variables, no API keys, no server.

- No edit / overrides

## Days columns

The reorder xlsx exposes two days fields:

- **Amazon Days** — Amazon's raw days-of-supply field.
- **Display Days** — our computed value (Available ÷ Weighted Velocity), and the one the upstream script uses to assign URGENT / PLANNED / UV / STEEL sections.

The dashboard drives everything off **Display Days**: the big number, the progress-bar width, and the red (<30) / yellow (30–45) / green (45+) color. The big number is rounded to an integer. **Amazon Days** is shown as small grey reference text beneath it (e.g. `16d` / `Amazon 20d`), and the full float appears in the cell's hover tooltip. The OUT OF STOCK badge is driven by the `Out Of Stock` flag column, unchanged.

## Optional: auto-load the newest file from Google Drive

The dashboard can load the most recent reorder xlsx straight from a Drive folder instead of a manual upload. This is **optional** — leave it unconfigured and the app works exactly as before (the Drive button stays hidden).

It uses browser-side Google Identity Services (OAuth token flow) + the Drive REST API. No backend, no stored secrets. The OAuth Client ID is public by design.

### One-time Google Cloud setup

1. Go to <https://console.cloud.google.com/> → create (or pick) a project.
2. **APIs & Services → Library →** enable **Google Drive API**.
3. **APIs & Services → OAuth consent screen:** set up an *External* (or *Internal* if Workspace) app. Add your Google account as a test user. Add the scope `.../auth/drive.readonly`.
4. **APIs & Services → Credentials → Create credentials → OAuth client ID → Web application.**
   - **Authorized JavaScript origins:** add your deploy URL(s) and `http://localhost:5173`.
   - Save and copy the **Client ID** (looks like `1234-abc.apps.googleusercontent.com`).
5. Open the Drive folder that receives the weekly Miami reorder files. The **folder ID** is the part after `/folders/` in its URL.

### Wire it up

```bash
cp .env.example .env
# then edit .env:
#   VITE_GOOGLE_CLIENT_ID=<your client id>
#   VITE_DRIVE_FOLDER_ID=<the folder id>
#   VITE_DRIVE_FILE_PREFIX=Reorder_Miami   # newest file whose name starts with this wins
npm run build      # env vars are baked in at build time
```

On Netlify, set the same three vars under **Site settings → Environment variables** and redeploy (no `.env` needed there).

### Behavior

- When configured, the empty state shows **"↧ Load latest from Google Drive"**. First click opens the Google consent popup; after that, the token is cached for the session.
- On return visits within the same session (valid cached token), the newest file auto-loads on open with no click.
- The newest file is chosen by Drive `modifiedTime` (descending), filtered to `.xlsx` whose name starts with `VITE_DRIVE_FILE_PREFIX`.
- Scope is `drive.readonly` — the app can only read, never modify Drive.
